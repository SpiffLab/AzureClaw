## ADDED Requirements

### Requirement: Subscription-scoped Bicep entrypoint
The repository SHALL provide `infra/main.bicep` as a `targetScope = 'subscription'` Bicep file that creates a single resource group named `${prefix}-${environment}-rg` and invokes every per-service module in `infra/modules/` to provision the resources inside that resource group.

#### Scenario: main.bicep compiles cleanly
- **WHEN** a contributor runs `bicep build infra/main.bicep --stdout`
- **THEN** the command exits 0 and emits a valid ARM JSON template

#### Scenario: main.bicep targets the subscription scope
- **WHEN** the file is inspected
- **THEN** the first non-comment line declares `targetScope = 'subscription'`

#### Scenario: main.bicep provisions exactly one resource group
- **WHEN** the file is inspected
- **THEN** it declares one `Microsoft.Resources/resourceGroups@*` resource
- **AND** every per-service module is invoked with `scope: rg` so all resources land inside that group

### Requirement: Per-service Bicep modules
The repository SHALL provide a Bicep module under `infra/modules/` for each Azure service AzureClaw uses: `cosmos.bicep`, `aisearch.bicep`, `keyvault.bicep`, `appinsights.bicep`, `servicebus.bicep`, `storage.bicep`, `containerapps.bicep`, `acr.bicep`, `contentsafety.bicep`, and `foundry.bicep`.

#### Scenario: Every module file compiles cleanly
- **WHEN** a contributor runs `bicep build infra/modules/<name>.bicep --stdout` for any module file
- **THEN** the command exits 0 and emits valid ARM JSON

#### Scenario: Every module declares a useful set of outputs
- **WHEN** a module is inspected
- **THEN** it declares `output` statements for the values that callers (other modules or `main.bicep`) need to wire dependent resources together — at minimum the resource id and name

#### Scenario: No module hard-codes secrets
- **WHEN** a module is inspected
- **THEN** no secret literal (API key, connection string, password) appears in source
- **AND** any value that would otherwise be a secret is either generated by Azure (e.g. via `listKeys()`) or accepted as a `@secure()` parameter

### Requirement: Cosmos DB containers match the AzureClawConfig schema
The `cosmos.bicep` module SHALL create a Cosmos DB account with the NoSQL API, autoscale enabled, a database named `azureclaw`, and the four containers `threads`, `checkpoints`, `audit`, and `sites` whose partition key paths match the names declared in `config.example.yaml` (`/session_id` for the first three, `/site_id` for `sites`).

#### Scenario: All four containers are declared
- **WHEN** the module is inspected
- **THEN** containers named exactly `threads`, `checkpoints`, `audit`, and `sites` are declared as child resources of the database

#### Scenario: Partition keys match config.example.yaml
- **WHEN** the module is inspected
- **THEN** the `threads`, `checkpoints`, and `audit` containers have `partitionKey.paths` containing `/session_id`
- **AND** the `sites` container has `partitionKey.paths` containing `/site_id`

### Requirement: AI Search index schema is committed alongside the service
The `aisearch.bicep` module SHALL create an Azure AI Search service and the `azureclaw-memory` index with at minimum these fields: `id` (key, retrievable), `session_id` (filterable, retrievable), `text` (searchable, retrievable), `embedding` (vector, dimensions = 3072 to match `text-embedding-3-large`), and `created_at` (filterable, sortable).

#### Scenario: Index is declared in the same module as the service
- **WHEN** the module is inspected
- **THEN** the `Microsoft.Search/searchServices@*` resource and the `Microsoft.Search/searchServices/indexes@*` child resource are both declared
- **AND** the index name is `azureclaw-memory`

#### Scenario: Vector field dimensions match the embedding model
- **WHEN** the module is inspected
- **THEN** the `embedding` field's `dimensions` is `3072`

### Requirement: Foundry deploys both chat and embedding models
The `foundry.bicep` module SHALL create an Azure AI Foundry project plus a chat-completions deployment of `gpt-5.4-mini` and an embeddings deployment of `text-embedding-3-large`. Both deployments SHALL be referenceable from `main.bicep` outputs so application code can resolve them at runtime via configuration.

#### Scenario: Both model deployments are declared
- **WHEN** the module is inspected
- **THEN** at least two `*/deployments@*` resources are declared
- **AND** one references the model `gpt-5.4-mini`
- **AND** one references the model `text-embedding-3-large`

#### Scenario: Deployment names are emitted as outputs
- **WHEN** the module is inspected
- **THEN** the chat deployment name and the embedding deployment name are declared as `output` statements

### Requirement: Key Vault grants Container Apps managed identity access
The `keyvault.bicep` module SHALL create the Key Vault and an access policy granting `get` and `list` permissions on secrets to the principal id passed in as a parameter. `main.bicep` SHALL pass the Container Apps environment's system-assigned managed identity principal id as that parameter.

#### Scenario: Access policy is parameterized
- **WHEN** the module is inspected
- **THEN** it accepts a `principalId` (or equivalently named) parameter
- **AND** the access policy's permissions are `get` and `list` on `secrets`

### Requirement: Container Apps environment is provisioned with the gateway and a sidecar slot
The `containerapps.bicep` module SHALL create a Container Apps environment, the `azureclaw-gateway` Container App (with a placeholder image such as `mcr.microsoft.com/k8se/quickstart:latest`), and reserve a sidecar container slot in the same Container App for the browser MCP server image that will be built and pushed by a later change.

#### Scenario: Gateway app references the environment
- **WHEN** the module is inspected
- **THEN** the gateway Container App references the Container Apps environment id

#### Scenario: System-assigned managed identity is enabled on the gateway
- **WHEN** the module is inspected
- **THEN** the gateway Container App declares `identity: { type: 'SystemAssigned' }`

#### Scenario: Sidecar slot is documented
- **WHEN** the module is inspected
- **THEN** the gateway Container App declares at least one container in `template.containers` for the gateway, and a comment or a second container entry documents the future browser MCP sidecar slot

### Requirement: azd project manifest at the repo root
The repository SHALL provide `azure.yaml` at the repo root that follows the `azd` schema and declares (a) the project name `azureclaw`, (b) the path to `infra/main.bicep`, and (c) the gateway service whose source path is `src/azureclaw/` and language is `python`.

#### Scenario: azd recognizes the project
- **WHEN** a contributor runs `azd config get` from the repo root after this change merges
- **THEN** azd loads `azure.yaml` without errors and reports the `azureclaw` project name

#### Scenario: Gateway service points at the Python package
- **WHEN** `azure.yaml` is inspected
- **THEN** the `services.gateway` block has `language: python` and `project: ./src/azureclaw`

### Requirement: bicep-validate CI gate becomes load-bearing
With this change merged, the `ci/bicep-validate` job in `.github/workflows/ci.yml` SHALL run `bicep build` for every `infra/**/*.bicep` file on every PR and SHALL fail the PR if any file does not compile.

#### Scenario: bicep-validate runs against every module on every PR
- **WHEN** a pull request is opened against `main` or `develop` after this change merges
- **THEN** the `ci/bicep-validate` job iterates over `infra/main.bicep` and every file under `infra/modules/`
- **AND** runs `bicep build` against each
- **AND** fails the job (and therefore the PR) if any compilation error is reported

## MODIFIED Requirements

### Requirement: Bicep what-if previews on infra PRs
When a PR touches files under `infra/`, the repository SHALL run `az deployment sub what-if` against the configured dev subscription and post the change-set diff as a PR comment. If the dev subscription is not configured, the workflow SHALL fall back to a no-op so external contributors are not blocked. The previously documented "skip when `infra/main.bicep` does not exist" branch is removed because that file is now guaranteed to exist on every commit on `main`.

#### Scenario: What-if posts a PR comment when subscription is configured
- **WHEN** a PR modifies a file under `infra/` and the `AZURE_DEV_SUBSCRIPTION_ID` repository variable is set
- **THEN** the workflow runs `az deployment sub what-if` against `infra/main.bicep` and posts the output as a PR comment

#### Scenario: What-if is a no-op when subscription is not configured
- **WHEN** a PR modifies a file under `infra/` and `AZURE_DEV_SUBSCRIPTION_ID` is empty
- **THEN** the workflow exits successfully without invoking the Azure CLI
